# Content Canvas — 生成迴路重新設計 規格

> 狀態：設計定案（2026-05-29，Andrew 確認）。施工依「落地順序」分階段，每階段可驗證。
> 這份取代「一個一個修按鈕」的做法。脊椎：**人掌方向與取捨，AI 在指定方向上做深度、有根據、會累積的生成。**

---

## 0. 設計脊椎（為什麼這樣設計）

1. **AI 不是腦機接口**——沒線索生不出有效內容。所以第一輪必然粗，價值在「會累積收斂的迴路」。
2. **單位是「一輪生成」，不是「做完一個節點」**——迭代是要的，挫敗來自「一輪內部按鈕太多」+「輪次之間線索不累積」。
3. **方向（發散/收斂）是使用者的選擇，不是 AI 推斷的**——AI 猜方向會猜錯一半。
4. **按鈕只為「人的決定」存在，不為「搬資料」存在**——搬資料一律自動、原子、共享。
5. **發散要深**——每個生成節點都深度推導（3-6 個，不是 20 個淺殼）。

---

## 1. 互動模型

```
使用者轉兩個盤： 方向(發散 / 收斂)  ×  尺度(此節點 / 整個計劃)
        │
   按一次「生成這一輪」
        │
   AI 吃進【累積線索 + 方向】→ 串完該串的環節 → 給高質量推薦（集中呈現，不用一塊塊找）
        │
   使用者整理： 採用 / 拋棄 / 連結 / 合併   ← 這些同時變成下一輪線索
        │
   再轉盤再按 → 一階段一階段累積、收斂
```

AI 永遠不自己決定方向、不自己定案。

---

## 2. 方向 × 尺度 的四格行為

| | **此節點** | **整個計劃** |
|---|---|---|
| **發散** | 給這支片更多深度角度 / hook / 切入點 | 並行深度生成 3-6 個延伸候選節點 |
| **收斂** | 砍冗角度、磨利留下的、聚焦一個主張 | 建議 merge / remove / 排序、整併重疊節點 |

四格今天分散在四個按鈕（擴寫 / suggestCausal / 連線 / 全局檢討）。重設計後收進「方向×尺度」這一個選擇。

---

## 3. 「一輪生成」串哪些環節（step-linking）

一按串完，**全是 AI 產出、不是人的決定**，所以不該各自一顆按鈕：

```
查事實(node.sources) → 推導 research → 分類 job/stage → 生成 hooks → 生成拍攝角度
```

留給使用者的決定（保留，但集中、低摩擦）：採用/拋棄整輪、挑哪個 hook、確認哪些角度、寫自己的意圖備註。

---

## 4. 線索累積模型（讓迭代真的收斂）

**問題（已驗證）：** 現在 `expand` 只吃 `topic/userNotes/job/stage`，不吃「上一輪採用/拋棄了什麼」。所以迭代不收斂，只是原地重生。（唯一雛形：hooks 重生吃 `hookDirection`。）

**改法：** 節點上維護累積線索，每輪生成全吃進去：

```js
node.clues = {
  userInput,        // 主題 + 使用者意圖/備註
  facts,            // node.sources：查證事實（查一次存著）
  adopted: [],      // 歷輪採用的（往這方向多走）
  discarded: [],    // 歷輪拋棄的（避開這方向）
  direction,        // 這一輪：發散 / 收斂
}
```

採用 = 「往這走」、拋棄 = 「不要這方向」，都回灌下一輪。每輪比上一輪線索更多 → 更準。

---

## 5. 深度發散的品質契約

- **數量有界：3-6 個候選**（對齊全域 UX 規則「相關項 3-6 個不塞 20 個」），不是噴一堆淺殼。
- **每個都深**：有事實根據（web search）、差異化角度、hook、拍法——可直接拍的程度。
- **並行生成**：3-6 個同時深挖（不是一個一個等 70 秒）。
- **禁止**：套模板、換關鍵字的淺殼（= 已禁的 stageTemplates，違反 CLAUDE.md 核心哲學）。

---

## 6. 整理動詞：連結 + 合併（發散後當場可做）

- **連結**：把相關候選串成系列 / 因果鏈（今天藏在「連線模式」）。
- **合併**：把重疊候選併成一個（收斂操作，今天藏在「全局檢討」的 merge）。

收進迴路：發散出候選後，當場就能連、能併，不用跳到別的功能。

---

## 7. 底層資料模型（結構性保證「共享 + 同時 + 一致」）

| 收斂成「只有一個」 | 內容 | 取代掉的重工 |
|---|---|---|
| **一次採集** `node.sources` | facts（日後 market/visuals）查一次存著、全端點共讀 | 每次 expand 重查 70s；review 腦補規格 |
| **一次扇出** `applyRound(node, picks)` | 採用某輪 → 被採用的東西**原子性**寫進所有該去的欄位（job/stage/cta/hooks/角度/關鍵字/備註） | 套用CTA + 採納 + 自動分類 三條分開的寫入路徑 |
| **一個序列化** `serializeNode(node)` | 每個端點（expand/review/brief/plan/ask）讀同一份完整節點 | 現在各搞各的 4 套序列化 |
| **一個指標來源** | readiness / completeness / health 都從同一組欄位算 | 三個指標互相矛盾 |

> 「哪些資料共享、哪些區塊同時拿到值」在這結構下不用逐個回答：一起推導的就一起寫、一起被讀，沒有任何一欄會比另一欄舊。

---

## 8. 按鈕收斂表

**Before（搬資料的按鈕，內耗來源）：** 擴寫、自動分類、套用CTA、採納研究、Hook精煉、全局檢討、suggestCausal、連線模式…

**After：**
- 主互動：**方向盤(發散/收斂) × 尺度(節點/計劃)** → 一按生成一輪
- 整理動詞：採用 / 拋棄 / 連結 / 合併
- 真正的決定：挑 hook、確認角度、寫意圖備註、生成 Brief（只組裝不重算）
- 移除：孤兒的 AI 潤稿（建立在已不存在的舊 6 欄 brief 上）

---

## 9. 落地順序（設計是一整塊，施工分三刀，每刀可驗）

> 受夠了 plausible-but-wrong：每刀動工前走前閘門三題，改完走後閘門完整複查，你確認才 push。

1. **`applyRound` 一次扇出 + 砍搬資料按鈕** — 最直接解「按四次」的痛。先做。
2. **`serializeNode` 統一 + 指標單一來源** — 端點一致，順手堵 review 腦補。
3. **方向盤 × 尺度 + 深度並行發散 + 連結/合併入流程** — 互動模型本體。
4. **`node.sources` 採集/推導分層 + clues 累積** — 收斂迴路 + 為 vision/YouTube 鋪路。

（1、2 是地基；3 是使用者最有感的互動；4 讓迭代真的累積。可按你要的順序調。）
